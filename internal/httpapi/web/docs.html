Safe operating procedures for private Nebula networks.

A practical guide for network operators, administrators, and read-only reviewers using the Mesh control plane.

Start with the rollout checklist Troubleshoot a node

Recommended path

Build a healthy network

Follow this order for a new environment. Each stage creates evidence needed by the next.

1

Create the network

Choose a private IPv4 range that does not overlap any site, cloud, container, or existing overlay range. Mesh creates and encrypts a dedicated Nebula certificate authority.

2

Deploy two lighthouses

Place lighthouses in different failure domains, give each a stable public UDP endpoint, enroll them, and wait until both are operational.

3

Enroll members

Create one enrollment per machine. Deliver the bootstrap trust anchor independently, run the supported installer on the target host, and never paste enrollment secrets into tickets or chat.

4

Verify before rollout

Run deployment readiness and review heartbeat and active-probe evidence. Test real overlay traffic before adding firewall, DNS, relay, or routed-subnet policy. Roll out each change to a small canary group first.

Operator guide

Networks and node roles

Mesh manages a Nebula trust domain, its signed configuration, and the lifecycle of every enrolled machine.

Network

A network owns one overlay CIDR, certificate authority, configuration-signing identity, firewall policy, DNS policy, relay policy, and route policy. Retiring a network destroys its active authority and is irreversible.

  • Use a unique RFC1918 range between /16 and /28.
  • Treat the network name and CIDR as long-lived identity.
  • Create separate networks when machines must not share trust.

Lighthouse

A lighthouse provides Nebula discovery and can also provide managed DNS or relay service when those features are enabled.

  • Use at least two operational lighthouses.
  • Place them in separate infrastructure failure domains.
  • Provide a reachable public UDP host and port when creating each lighthouse, for example vpn.example.com:4242.
  • Expose the configured UDP port through every cloud and host firewall.

Member

A member is an ordinary endpoint or server in the overlay. Members can belong to security groups and may optionally advertise routed subnets.

  • Groups control policy targeting; site and failure-domain labels control readiness placement.
  • A gateway is a member with routed-subnet ownership.
  • Node names should remain stable and meaningful to operators.

Use the operations workspace

The control plane has a fleet view and a separate workspace for each network. Review fleet alerts first, then open the affected network.

  • Networks shows the fleet overview, rollout convergence, and network directory. Expand Needs attention to read the full alert list.
  • Open a network row to see its topology and next setup action. Manage nodes contains the detailed lifecycle controls.
  • Within a network, use Settings for readiness, DNS, relays, firewall, routing, certificate authority, and retirement. The signed-in Viewer, Operator, or Admin role controls which actions are available.
  • Activity shows append-only security events. It also includes recovery-code administration when the feature is enabled and the signed-in role has permission.
  • On smaller screens, use the labeled bottom navigation. Keyboard users can skip directly to the control-plane content.

Enrollment and lifecycle

An enrollment can be used once. The target generates its private key locally, and Mesh returns signed, versioned runtime configuration.

Enroll a machine

Create the node in Mesh, install one authenticated Mesh runtime release on the intended machine, run the one-time enrollment command, and activate the managed agent and Nebula service.

  1. Create a node with the correct role, site, failure domain, endpoint, groups, and routed subnets.
  2. Transfer the independent bootstrap trust anchor through an operator-approved channel.
  3. Authenticate the platform bootstrap installer, then install either the exact signed online release selected for the host OS and architecture or the complete signed offline snapshot.
  4. Run the displayed enrollment command with nebula, nebula-cert, and meshctl from that one installed release.
  5. Confirm the agent and Nebula service are active.
  6. Verify the node is operational and can exchange real overlay packets.
Protect enrollment material

An enrollment token is a short-lived secret for exactly one node. Reissue it if delivery is uncertain; the previous token becomes invalid.

Runtime prerequisite and trusted installation

Before production enrollment reads a token or changes local state, Mesh checks nebula and nebula-cert. Both files must be executable, report the same supported version, and resolve beside meshctl in one authenticated installed release.

  • On Linux, use mesh-install. On another supported platform, use its Mesh installer. Online installation selects the OS and architecture artifact authorized by threshold-signed release metadata, then verifies its signed size and SHA-256 before installation.
  • Do not fetch GitHub's moving latest Nebula release during enrollment. That would bypass Mesh release-root authorization, compatibility policy, package evidence, anti-rollback state, and recovery.
  • For an offline or air-gapped host, independently authenticate the bootstrap verifier and installer, then install the complete root-private signed snapshot before enrollment.
  • If either runtime is missing, old, mismatched, in another directory, or not executable, Mesh leaves the token unread and does not start enrollment. Follow the mesh-install guidance in the error, then rerun the same enrollment command.
  • Development builds can exercise the version-pair check without claiming production package provenance. Production acceptance still requires the supported installer and its platform proof gates.

Replace or recover a node

Use agent recovery when the host identity is intact but its control credential is lost. Use identity replacement when the private key or host identity may be compromised.

  • Recovery rotates the agent credential and keeps the existing certificate identity.
  • Identity replacement revokes the old node and creates a new one-time enrollment with a new overlay address.
  • Certificate rotation issues a new certificate for the existing node identity.
  • Archive only after revocation and certificate-safety checks permit it.

Revoke a node

Use Revoke for a permanent trust cutoff when an active machine is retired, lost, or no longer authorized. Only an administrator with node-security permission can perform this irreversible operation.

  • Mesh invalidates the node's enrollment, agent, and recovery credentials.
  • Every still-valid certificate issued to the node is added to the signed Nebula blocklist through its recorded expiry.
  • Relay and firewall-canary assignments are removed safely, routed-subnet ownership is released, and the network advances exactly one signed configuration revision.
  • Keep every peer on supported Nebula 1.10.3 or newer; older vulnerable releases do not meet Mesh's blocklist security boundary.
  1. Cancel a never-enrolled pending node instead. Use Rotate certificate for a routine same-key replacement, or Replace identity when the private key is lost or may be compromised.
  2. Finish or cancel any route transition involving the node; Mesh refuses revocation while that node participates in a route transfer or route-profile change.
  3. Open the network, locate the active node, choose Revoke, review the permanent impact, and confirm the selected node. Mesh binds the request to the exact node, current signed revision, and one idempotency key.
  4. Wait for the node to show Revoked and for operational managed peers to apply the new configuration revision. If the response was interrupted, use Verify revocation to replay the exact request without performing a second trust cutoff.
  5. From an updated managed peer, verify that the revoked machine can no longer establish a Nebula handshake or exchange application traffic. A heartbeat or inventory status alone is not packet-level proof.
  6. After the latest certificate expiry plus the five-minute safety margin, use Archive record to remove expired issuance and blocklist history. Archival does not undo revocation; reuse requires a fresh node ID and one-time enrollment.
Revocation cannot be undone

Do not use revocation as a temporary disable switch. A stale, unmanaged, or fail-open peer that has not applied the signed blocklist may continue trusting the old certificate; update or isolate that peer before treating the cutoff as complete.

Health and heartbeat meaning

A fresh authenticated heartbeat proves lifecycle contact with the managed agent. It does not by itself prove that arbitrary peers can exchange application traffic.

  • Operational means desired signed state, certificate generation, agent status, and Nebula runtime have converged.
  • Active probes provide stronger evidence but still cover only the configured probe relationship.
  • Use real packet and application checks for production acceptance.

Firewall rules and groups

Mesh compiles Nebula firewall policy from explicit node and group selectors, then distributes the signed result to managed nodes.

Design policy

Start with the smallest required flows. Prefer group selectors for stable roles and node selectors for narrow exceptions.

  • Every node belongs to the built-in all group.
  • Use named groups such as app, database, admin, or monitoring for reusable policy.
  • Preview effective policy before deployment to confirm target-node, target-group, and peer-node expansion.
  • Keep management access separate from application traffic.

Roll out safely

Use a canary rollout for material firewall changes. Mesh can pause, resume, promote, or roll back the in-flight policy.

  1. Preview and review the effective policy.
  2. Select representative canary nodes.
  3. Start the rollout and test both traffic that should pass and traffic that should fail.
  4. Pause if investigation is needed, or roll back if behavior is incorrect.
  5. Promote only after canaries and health evidence are current.

Change node groups

Group changes rotate the node certificate so group membership is cryptographically bound to the new identity claims.

Expect certificate convergence

Do not treat the group change as complete until the node reports the desired certificate generation and remains reachable under the new policy.

DNS, relays, and routed networks

DNS, relays, and routing change how traffic finds or reaches peers. Enable one feature at a time and test real packets after each revision.

Managed overlay DNS

Mesh can publish node records through selected operational lighthouses. Members receive the signed resolver list as part of network configuration.

  • Keep the standard DNS port unless the environment requires an approved alternative.
  • Permit DNS traffic in the Nebula firewall.
  • Query every selected resolver directly before depending on names.
  • Linux native split DNS is an optional host integration; canary it and confirm the overlay interface exists before wider rollout.
Resolver failure is fail-closed

A managed native-resolver apply failure can quarantine the runtime until signed configuration is corrected. Test on one non-critical node first.

Nebula relays

Mesh supports Nebula's native relay capability. Selected operational lighthouses act as relay servers, while clients continue to prefer a direct peer path and use a relay when direct UDP connectivity is unavailable.

  • Enable relays under network settings and select one or more active lighthouses.
  • Allow client UDP traffic to the relay lighthouse endpoint.
  • Relayed packet contents remain encrypted end to end; the relay can observe routing and connection metadata.
  • Test relay behavior by blocking only the direct peer path, not the lighthouse path.
  • Treat upstream relay support as an advanced feature and retain redundant direct connectivity where possible.

Routed subnets

A routed-subnet owner forwards traffic between the Nebula overlay and a non-Nebula network behind that node.

  • Enable operating-system forwarding and downstream return routing.
  • Declare the complete routed-subnet set on the gateway node.
  • Use the route-transfer workflow to move ownership without conflicting advertisements.
  • Use weighted route policies for deliberate ECMP distribution across multiple owners.
  • Test forward and reverse traffic from a real host behind the gateway.

Identity and security operations

Authentication decides who can use the control plane. Signed network state and certificate lifecycle decide what enrolled machines can do.

Roles

OIDC groups map to one of three Mesh roles.

  • Viewer: read network, health, readiness, runtime telemetry, and audit information.
  • Operator: viewer access plus routine network and node changes.
  • Admin: operator access plus certificate authority, node-security, recovery, session, and retirement operations.

Use Mesh Desktop

Mesh Desktop is a preview, unprivileged operator console for Linux and Windows. It connects to a remote Mesh control plane. It does not install Nebula, manage a local tunnel, or run the privileged node service.

  • Mesh Desktop keeps the session and CSRF cookie pair in the operating system credential store. A legacy administrator token, when a deployment still allows one, stays in memory and is not saved.
  • Pending, denied, and expired requests return no session credentials. An approved request can create a session only once.
  • The app must wait for the server-provided polling interval. A 429 response includes Retry-After and means the client polled too soon.
  • The preview can create networks and node enrollments, reissue a pending enrollment, rotate or permanently revoke a node identity, revoke a control-plane session, and create one-use recovery access. Every action is checked again by server-side RBAC.
  • A lighthouse enrollment requires its reachable public UDP endpoint. Enrollment tokens and recovery codes appear in a one-time custody view and are removed when the view closes or the app leaves the foreground.
  • Linux Debian and Windows MSIX artifacts produced by the current CI workflow are unsigned. Verify the published checksum, then sign and validate the package in a protected release environment before distribution.
  • Use the web control plane or API for any workflow that the preview desktop client does not yet expose. The server remains the authorization authority for every request.
  1. Enter the exact HTTPS origin of the Mesh control plane. Plain HTTP is limited to an explicitly enabled loopback development profile.
  2. Choose browser sign-in. Mesh Desktop creates a request that expires after five minutes and opens the system browser. The verification URL contains the public request ID, not the poll secret.
  3. Sign in to the same control plane if needed. Approve the request only if you started it from Mesh Desktop. The browser also lets you deny the request.
  4. Return to Mesh Desktop. After approval, the server creates a separate Mesh session with the same principal, role, and permissions as the browser session. The browser does not give its password or session cookie to the app.
  5. Check the role shown in Mesh Desktop before making changes. Sign out when the workstation no longer needs access. An administrator can also revoke the session from the session inventory.
Update every control-plane replica before restoring writes

This release adds desktop_authorizations to the shared identity document. Older strict readers reject that field after a new server writes the document. During a multi-replica upgrade, block identity-writing traffic, replace every control-plane replica, then restore traffic. After the field has been written, do not roll back to an older build unless you first use an operator-reviewed state migration or restore a verified pre-upgrade identity backup.

Rotate the network CA

CA rotation is a staged trust transition. Keep the workflow open and allow every active node to converge before advancing.

  1. Prepare dual trust.
  2. Wait for every active node to apply the dual-trust revision.
  3. Activate the replacement CA and wait for replacement certificates.
  4. Finalize replacement-only trust after old-root dependencies are gone.
  5. Complete the rotation and verify final packet connectivity.
Never skip convergence gates

Offline or stale nodes cannot satisfy a safe trust transition. Restore or explicitly revoke them before finalizing.

Sessions and recovery access

Admins can inspect and revoke control-plane sessions created by the web app or Mesh Desktop. If the deployment enables one-use recovery codes, create and custody them before an identity-provider outage.

  • Revoke unfamiliar or unneeded sessions immediately.
  • Store recovery codes outside the Mesh host and identity provider.
  • A displayed recovery code cannot be retrieved again.
  • Mesh Desktop creates recovery access with a seven-day lifetime. Use the web control plane or API when a different expiry is required.
  • Keep normal OIDC access protected with the organization's MFA policy.

Use the API reference

Mesh publishes its OpenAPI 3.1 contract at /openapi.json and a searchable reference at /api-docs.html. Each operation lists its RBAC permission, availability, parameters, JSON schemas, error responses, placeholder examples, and curl command.

  • Sign in normally before using interactive calls. The page uses the current same-origin browser session and never asks for or stores a legacy administrator bearer credential.
  • For a state-changing browser call, the page reads the Mesh CSRF cookie, sends X-Mesh-CSRF, and uses the browser's exact Origin. The server returns 403 if the cookie, header, origin, or JSON content type is missing or does not match.
  • Agent, enrollment, recovery, and other credential-bearing calls are intentionally not executable in the browser reference. Run their placeholder curl examples only from the intended client with an approved secret-delivery path.
  • Read the current resource and revision before a mutation, preview policies where supported, review the exact request, confirm that it changes live state, then verify the returned state and health evidence.
  • Treat 401 as missing or invalid authentication, 403 as permission or browser-request proof failure, 409 as stale revision or invalid lifecycle state, and 429 or 503 as a bounded service condition to retry later.
Interactive requests change the live control plane

The API reference is not a sandbox. Eligible POST, PUT, PATCH, and DELETE operations require an explicit confirmation and are sent to the same Mesh origin that served the page.

Readiness, backup, and retirement

Readiness reports signed state, placement, and packet evidence separately. A green process check does not prove that peers can exchange traffic.

Deployment readiness

Readiness checks topology, lighthouse diversity, placement labels, endpoint syntax, overlap evidence, DNS evidence, and lifecycle state from one bounded snapshot.

  • Assign a site and failure domain to every active node.
  • Use at least two lighthouses in different failure domains.
  • Resolve and test every public lighthouse endpoint externally.
  • Treat verification_required as missing evidence that must be resolved or documented, not as success.

Backup and restore

The JSON storage backend uses an authenticated offline backup. The server must be stopped so the utility can take both state locks and verify bound credentials.

  1. Generate and independently custody a 256-bit backup key.
  2. Stop every Mesh writer.
  3. Create and verify a fresh archive with the exact current utility version.
  4. Record the authenticated backup ID in an external monotonic catalog.
  5. Restore only into a nonexistent target directory and validate before starting a server.
Backups are versioned operations

Build and package mesh-backup from the same approved source version as the server. A stale operator binary must not be used against a newer store schema.

Retire a network

Retirement permanently removes encrypted CA and signing material, nodes, credentials, enrollments, and runtime telemetry for the trust domain.

  • Confirm the exact current configuration revision and case-sensitive network name.
  • Stop unmanaged or fail-open Nebula processes before retirement.
  • Expect every managed agent credential to become unauthorized immediately.
  • The retired name and overlapping CIDR remain reserved to prevent trust-domain confusion.
Retirement cannot be undone

Use verified backups for disaster recovery. Do not use retirement as a temporary disable switch.

Troubleshooting checklist

Start with the failing layer. Change only one of firewall, routing, DNS, or certificates at a time.

Node is offline or stale

  • Confirm mesh-agent and mesh-nebula are active on the host.
  • Check the last authenticated heartbeat and applied configuration revision.
  • Verify the host clock, public HTTPS trust, and access to the control plane.
  • Confirm the node certificate and agent credential have not expired or been revoked.
  • Use agent recovery only when identity replacement is not required.

Peers cannot connect

  • Test the lighthouse public UDP endpoint from outside its network.
  • Check cloud, host, and Nebula firewall policy independently.
  • Establish outgoing traffic once to populate dynamic peer mappings.
  • Compare direct and relay behavior without blocking lighthouse traffic.
  • Use packet captures and Nebula logs without exposing private keys or bearer credentials.

DNS or routed traffic fails

  • Query each managed DNS resolver directly by overlay IP.
  • Confirm the native resolver points at an existing overlay interface.
  • Check route ownership, forwarding, downstream return routes, and effective MTU.
  • Test both directions with a real non-Nebula host behind the gateway.
  • Roll back the last isolated policy change before combining fixes.
