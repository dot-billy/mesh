## Trust boundaries

Mesh is an online certificate authority. A control-plane compromise can issue identities for every network whose CA it holds. Protect the host, master key, admin credential, state backups, TLS endpoint, and update path as sensitive infrastructure.

The encrypted recovery archive is equivalent to the live control plane once paired with its separately held backup key: it contains the master key, administrator bearer, both state stores, every encrypted network CA/signing key, and persisted browser sessions. Keep archive storage and backup-key custody in separate access and failure domains.

Enrollment is trust on first use over HTTPS. The first verified response pins both the network's Ed25519 configuration-signing public key and exact Nebula CA digest. A TLS compromise during that first exchange can choose those trust anchors. After enrollment, a TLS terminator or API transport compromise without the signing key cannot replace the CA, certificate, public key, expiry, identity, or configuration while replaying an otherwise valid signed artifact. Enrollment, renewal, and agent recovery parse and re-marshal exactly one canonical Nebula public-key PEM before hashing or signing it; a second PEM, comment, or other remainder is rejected instead of being ignored by the upstream CLI parser.

Mesh never receives a node private key. `meshctl` generates the Nebula keypair and a 256-bit scoped agent bearer locally. Enrollment sends only the Nebula public key and SHA-256 agent-token hash.

## Control-plane key and state custody

Every network has a separate Nebula CA key and Ed25519 artifact-signing key. Both private keys are encrypted with AES-256-GCM under `MESH_MASTER_KEY`, with versioned purpose-specific associated data. Startup decrypts each signing key, derives its public key, and fails closed if the persisted pair is incomplete or mismatched.

The JSON store is single-writer locked. State updates use a mode-private temporary file, file `fsync`, atomic rename, and parent-directory `fsync`. The PostgreSQL preview instead stores the same two exact encoded documents as bounded checksummed `BYTEA` values with revisions and write receipts. Both backends reject unknown fields and invalid graphs, and every startup and mutation validates identity, uniqueness, network membership, lifecycle counters, and record references before CA use. CA plaintext is held only while invoking `nebula-cert` and its byte buffer is cleared afterward. Agent and enrollment credentials are persisted server-side only as SHA-256 hashes.

The ordinary control graph shares recovery's 128 KiB CA/host-certificate and 4 KiB CA-private-key contracts; encrypted limits are derived from the plaintext ceiling plus AES-GCM nonce/tag and canonical raw-base64 expansion. Config-signing public/private ciphertexts must have their exact Ed25519/AES-GCM encodings. Signed managed configurations are nonempty UTF-8 without carriage returns and share one 4 MiB generation/intake ceiling with the node agent. Node groups are grammar-checked, strictly sorted, unique, include `all`, and are capped at 64. Public endpoints use a bounded DNS/punycode or IP host and a numeric UDP port. Enrollment and renewal claim metadata uses the same bounded identifier/hash grammars as live credentials, and every retained recovery result revalidates its embedded node, certificate, configuration, and credential lifecycle. Revocation reasons and audit action/resource/detail keys and scalar values reject invalid UTF-8, control text, excessive count/depth, and non-JSON types. These per-record limits complement rather than replace the 64 MiB aggregate control-document ceiling; top-level history and inventory collections are not assigned arbitrary cardinality caps.

Control schema v3 gives every node canonical lowercase `site` and `failure_domain` labels. They are operator assertions used only for inventory grouping and readiness diversity; they do not enter Nebula certificates, signed configuration, or firewall authorization. Security groups remain the only one of these metadata sets used by Nebula policy. The one-way v2-to-v3 migration marks every existing node `unassigned` instead of inventing locality or independence. An administrator can update placement without reissuing a certificate or changing the signed configuration revision, and the mutation is audited. Consequently, an administrator or state owner can falsify diversity: readiness is an operational planning signal, not independent proof of physical separation.

Control schema v4 makes every network's Nebula lighthouse DNS state explicit. The one-way v3-to-v4 migration records disabled DNS on canonical port 53, adds one audit event, and does not advance a signed configuration revision because it renders no new bytes. Enabling requires a canonical UDP port from 1 through 65535 that differs from the network's Nebula listener port, and the current inbound policy must cover that port for every managed node through `group: all`, `host: any`, or the complete network CIDR. DNS enable and firewall preview/deploy both reject a state that would remove that coverage. Signed configs set `serve_dns` only for lighthouses and bind each listener to that node's exact overlay IP; member configs never contain the listener. The API and dashboard expose only active lighthouse overlay resolver addresses. Nebula's responder learns certificate names from authenticated peers and is not a recursive resolver or forwarder. This responder setting alone never changes the host resolver; native integration is a separate explicit v12 control described below.

Control schema v5 makes every network's managed relay selection explicit. The one-way v4-to-v5 migration records disabled relay settings with an exact empty node-ID array, adds one audit event, and leaves every signed configuration revision unchanged because disabled settings render no bytes. Enabling requires 1–8 unique pending or active managed node IDs from that network. Selected nodes receive `am_relay: true` and do not use other relays; every non-selected node advertises only selected nodes that are active. A pending selection therefore cannot be exposed as an active relay address. Revoking a selected relay removes it atomically, disables relay settings when the selection becomes empty, and advances the network revision through the existing revocation rollout. The strict API and dashboard expose node identity, overlay address, role, and selection state, but no underlay endpoint or fabricated reachability evidence. Slack marks relay support experimental. Mesh does not measure relay capacity or prove public/private underlay reachability from both peers; placement and edge access remain operator responsibilities.

Control schema v6 binds every issued host certificate to its exact authoritative CA digest and adds a crash-durable per-network rotation state machine. The one-way v5-to-v6 migration derives that digest from the existing CA, records one audit event, and leaves configuration revisions, timestamps, identity state, and telemetry state unchanged. Planned rotation is gated through four explicit stages: `prepared` deploys old-plus-new trust; `rotating` signs same-key early renewals with the replacement CA; `finalizing` removes old trust only after every active node reports the target CA, authoritative certificate generation, fingerprint, and stage revision; `complete` clears transition metadata after replacement-only trust converges. A prepared rotation can be aborted before activation. Every trust-changing action is rejected while a replayable agent-recovery result is live, and every action requires the exact current configuration revision. Stable artifacts retain the v3 signed envelope for upgrade compatibility; transition artifacts use v4 to authenticate the previous trust digest and mandatory-renewal bit. Legacy agents therefore continue operating before rotation but fail closed during prepare and cannot satisfy activation. Only systemd-managed agents can provide authoritative runtime acknowledgment; PID/HUP and no-reload modes remain deliberately non-converged.

Control schema v9 adds a crash-durable per-network routed-subnet transfer state machine after the v8 firewall pause schema. The one-way v8-to-v9 migration writes an empty transfer receipt and one audit event without changing signed configuration bytes, revisions, or timestamps. The existing firewall rollout remains as described below; CA rotation and firewall rollout starts are fenced while a route transfer is active, and participating node revocation, replacement, and administrator certificate rotation are also fenced until the transfer reaches a terminal receipt.

Control schema v10 adds a crash-durable per-network active-owner route-profile edit state machine after the v9 route-transfer schema. The one-way v9-to-v10 migration writes an empty edit receipt and one audit event without changing signed configuration bytes, revisions, timestamps, or telemetry. Active route-profile edits are mutually exclusive with route transfers, CA rotation, and firewall rollout. Owner revocation, replacement, administrative certificate rotation, and direct firewall changes are fenced until the edit reaches a terminal receipt.

Control schema v11 adds durable per-network route policies after the v10 route-profile schema. The one-way v10-to-v11 migration writes empty route-policy state and one audit event without changing signed configuration bytes, revisions, timestamps, or telemetry. A stored policy must exactly match every current active certificate owner of one canonical prefix; route-profile promotion/removal, revocation, and enrollment activation reconcile membership while preserving surviving weights and controls. Policy writes are fenced during CA rotation, firewall rollout, route transfer, or route-profile editing.

Control schema v12 adds explicit native split-DNS state to the existing managed-DNS document. The one-way v11-to-v12 migration writes `native_resolver=false` and an empty search domain with one audit event, without changing signed configuration bytes, revisions, timestamps, or telemetry. Enabling requires managed lighthouse DNS plus one lowercase canonical domain other than `.local`; disabling managed DNS also clears native resolver state. Each node receives a strict, signed `mesh-native-dns-v1` comment containing only its overlay address, managed CIDR, chosen suffix, bounded resolver port, and sorted active-lighthouse overlay addresses. The agent starts a UDP adapter on that node address, verifies local-source single-question queries and exact upstream response tuples, forwards only the chosen suffix to authenticated lighthouse resolvers under bounded deadlines, rewrites only the exact answer owner, and never reads or forwards the host's ordinary resolver configuration. Linux registers that address and suffix transiently on the unique Mesh interface through systemd-resolved with `default-route=false`, DNSSEC/DoT/LLMNR/mDNS disabled for the link. Windows binds the adapter to UDP 53 and creates one local-machine NRPT suffix rule through the system PowerShell DnsClient module. It rejects a foreign exact-namespace owner, tags and tracks only the exact Mesh rule, proves configured and effective nameserver/namespace/encoding/DNSSEC fields, adopts successful add/remove operations after response loss, removes restart orphans, closes the adapter before cleanup, and restores the prior signed policy after a failed update. Partial application, disable, staleness, shutdown, interface replacement, or quarantine revert the owned resolver state. A current operational heartbeat must explicitly confirm native DNS is active when desired; an old or failing agent therefore cannot create false convergence. This mechanism assumes trusted host root and signed Mesh authority plus systemd-resolved and `resolvectl` on Linux or the DnsClient/NRPT stack on Windows. It is not a general recursive resolver. Linux has live proof; Windows remains gated on the clean-host native packet/cleanup receipt. macOS is not implemented.

Control schema v8 extends the crash-durable per-network firewall-rollout state machine with explicit pause/resume. The one-way v6-to-v7 migration records stable rollout state; v7-to-v8 adds a zero pause value. Each writes one audit event without changing signed configuration bytes, revisions, or timestamps. Starting a rollout selects 1–16 active canary nodes and retains the current policy as the exact rollback target; only selected nodes render the proposed target policy. Promotion remains unavailable until every selected node reports the exact staged revision, its own desired configuration digest, authoritative certificate generation and fingerprint, and a running managed Nebula process. Pausing advances one signed revision and returns selected canaries to the retained policy while preserving the target and cohort. Resuming advances again, re-delivers the target, and requires fresh exact convergence. Rollback remains available throughout canary and paused phases. Direct policy changes and DNS, relay, and CA transitions that could invalidate staged evidence are fenced while a rollout is active. Revocation remains available: it removes a selected canary atomically in either phase, and revoking the final selected canary automatically rolls back. Two closed automatic guards exist. If candidate validation or local runtime activation fails, the agent first restores its previous bundle, then may submit only the attempted signed revision, per-node digest, and closed `activation_failed` code. Separately, the ordinary monotonic heartbeat may trigger rollback only when a selected active canary reports the exact current target revision/digest, exact authoritative fingerprint and explicit certificate generation, `degraded`, and `nebula_running=false`. The server independently re-renders the current desired artifact and atomically advances one retained-policy revision. Paused, stale, forged, replayed, missing, prior-bundle, generation-free, healthy-contradictory, and generic running-degraded evidence cannot trigger either path. Audits and the strict dashboard expose only the node and bounded reason, never the raw local error. Every administrator or node-agent transition is attributed. Broader multi-sample and configurable health policies remain separate work.

Development secret generation is serialized by the state lock. Existing paths must be bounded, owner-controlled, private regular files; new 256-bit base64url values are created exclusively and fsynced.

For JSON, use `mesh-backup` for coordinated recovery rather than copying either domain file independently. The encrypted archive contains the exact master key and both state files; its separate backup key is the recovery authority. Losing both the live master key and a recoverable archive makes CA and signing keys unrecoverable, while possession of an archive and its backup key grants the full authority described above. PostgreSQL import can consume that archive for cutover, but ongoing PostgreSQL recovery requires database backup/WAL controls that are not yet production-proved.

## PostgreSQL storage and transport boundary

PostgreSQL is a second authority boundary, not merely an availability component. It contains token hashes, certificates, session and audit records, encrypted CA/signing material, and operational metadata. SHA-256 constraints and receipts detect accidental corruption and ambiguous application commits; they do not protect against a database owner who can replace a document, checksum, receipt, and provenance together. Protect the database service, administrative roles, replication/WAL backups, monitoring, and recovery approvals as sensitive control-plane infrastructure.

The migration, import, and runtime roles must be separate. Only the migration role owns the fresh dedicated database, `mesh` schema, `pgcrypto` extension, and tables. Because PostgreSQL 17 installs trusted-extension member functions under the bootstrap superuser, the mandatory cluster-admin ceremony must transfer only the `pgcrypto` function members to `mesh_migrate` before it revokes PUBLIC function execution and grants only `mesh.digest(bytea,text)` to import/runtime. Base migration/schema inspection permits that staging boundary; import fails before writing and runtime readiness fails closed until exact member ownership and execution privileges pass. The import role may select and insert the initial state/receipt/provenance rows but may not update, delete, or run DDL. The runtime role may select, insert receipt rows, and update only the revision/body/checksum/receipt/timestamp columns of the two existing documents; it may not insert state/import metadata, delete, prune receipts, or run DDL. The exact reference transfer, grants, and deployment proof are in the [PostgreSQL storage guide](ha-storage.md#least-privilege-roles).

PostgreSQL credentials are accepted only from a stable effective-user-owned, single-link mode-`0400` or mode-`0600` file at a clean absolute symlink-free path. The loader rejects ambient `PG*` settings, arbitrary runtime parameters, malformed/duplicate settings, unbounded connection/pool controls, and errors that could echo the DSN. The parsed pool necessarily retains its database credential in process memory, so OS/process/crash-dump protection remains required. Never place a DSN in arguments, environment variables, logs, or diagnostic bundles.

Production requires hostname-authenticated TLS on every primary and fallback route: `sslmode=verify-full`, `sslrootcert=system`, and `target_session_attrs=read-write`. A private database CA must be installed into the service account's operating-system trust store. Custom root files and client-certificate/mTLS configuration are currently rejected. Plaintext is accepted only behind the explicit `--allow-local-plaintext-postgres` development/test flag and only for numeric loopback addresses or a clean absolute Unix-socket directory; `localhost`, private LAN routes, and mixed fallbacks fail closed.

JSON-to-PostgreSQL import accepts only a Linux-stably-read, authenticated bound control-v2 through control-v13 backup and an independently supplied exact backup ID. It inserts both revision-1 documents, receipt evidence, and immutable provenance in one transaction and never overwrites nonempty state. Older documents remain byte-exact through offline verification and perform any remaining ordered migrations, including v11-to-v12 native DNS and the signed-byte-preserving v12-to-v13 firewall-scope boundary, on the first current-server startup. A lost commit acknowledgment or later validation failure can mean the import is already durable: keep writers stopped and run `mesh-storage verify`, never blindly retry. Verification is an import-era cutover check, not a PostgreSQL backup/PITR mechanism. Every application replica must still receive the same external `MESH_MASTER_KEY` and `MESH_ADMIN_TOKEN`; neither belongs in the DSN or database.

The current PostgreSQL proof includes a single-instance two-application-replica smoke, a bounded PostgreSQL 17 synchronous physical-standby promotion drill with named-standby `remote_apply`, a deterministic test-only ambiguous-commit drill across that promotion and a distinct writable authority, a bounded single-primary base-backup/archived-WAL recovery drill, an isolated Linux role/system-trust-path TLS drill, and an audited [maximum-valid-document](postgres-max-document.md) pass. The recovery drill writes create-only, fsynced but unauthenticated same-workspace no-writer evidence binding source identity/timeline, base and WAL digests, named point/LSN, exact API state, and the complete document/receipt/import manifest. Independent early and later authorities recover through read-only inputs, advance timelines, match before new writes, prove point-selected session absence/presence, and complete a recovered real-Nebula enroll/sign/verify/revoke lifecycle. The role/TLS drill proves exact grants and denials, hostname and fallback verification, hostnossl rejection, import-login disablement, password rotation, old-password rejection, and saved-diagnostic database-secret scanning using conventional minimal Linux trust paths. These bounded proofs do not establish sustained mixed writes, automatic election/fencing, production WAL retention/timeline-history custody, concurrent or failover-adjacent recovery, long-duration load/soak budgets, package-managed target-host trust, managed secret/certificate lifecycle, crash-collector safety, or distributed pre-authentication limiting. Until those gates pass, treat PostgreSQL as a preview rather than production HA or production PITR.

## Enrollment and node-side secrets

One-time enrollment tokens expire after 30 minutes and are atomically claimed. The server accepts an identical retry to recover an ambiguous response but rejects a changed public key or credential hash. After a successful legacy browser sign-in, the submitted administrator token is cleared from the input. Closing either one-time enrollment disclosure (including reissue) or agent-recovery disclosure clears its token, generated commands, and identifying display fields from the DOM; the operator must save the credential before continuing.

Before a new enrollment generates keys or writes its provisional journal, `meshctl` sends the one-time token only to the rate-limited, query-free `POST /api/v1/enroll/preflight` endpoint. Unknown, expired, used, or non-pending credentials all fail as unauthorized. The read-only response is `Cache-Control: no-store`, does not append audit history, and contains exactly the target role, canonical Mesh CIDR, a sorted unique set of at most 64 active lighthouse endpoints (plus the target itself when it is a lighthouse), and the token expiry. It contains no node address, groups, CA, signing key, certificate, control-plane resolver result, or private topology.

The future host strictly validates that response and its expiry. On Linux it reads the bounded kernel IPv4 route inventory and rejects any non-default route intersecting the Mesh CIDR; other platforms report route inspection as unsupported rather than fabricating a pass. It deduplicates canonical DNS names, skips IP literals, and requires every planned lighthouse name to resolve through the host's own resolver within one bounded three-second window. A member with no active lighthouse is rejected. Resolver errors, addresses, and local routes remain local. These checks occur before key generation, agent-bearer generation, journal creation, and the consuming enrollment POST, so a failed check leaves the same token reusable. The result is point-in-time route/DNS evidence only: it does not reserve the control graph against a concurrent administrator change, send UDP, or prove endpoint reachability. Exact signed configuration validation and current post-enrollment runtime readiness remain separate fail-closed checks.

An administrator can reissue enrollment only for a never-enrolled pending node. Reissue removes every prior token for that node and commits a globally unique replacement in the same transaction. An in-flight old claim loses its final compare-and-commit, and the raw replacement is returned once without server-side persistence. Active and revoked nodes are rejected.

An administrator can instead cancel a never-enrolled pending node through the query-free `POST /api/v1/nodes/{nodeID}/enrollment/cancel` operation. The request must carry the exact case-sensitive node name. One transaction removes the node and every enrollment record, releases its overlay address and routed-subnet reservations, removes any pending relay assignment, and appends an attributed `node.pending_cancelled` audit receipt; it does not create a revocation because no certificate identity exists. Removing a relay selection advances the signed network revision exactly once, while an otherwise unreferenced pending cancellation leaves signed configuration unchanged. Reuse of the human name, address, or route creates a fresh random node ID and one-time credential.

Enrollment claim/signing occurs outside the store lock, but its final commit requires both the exact enrollment record and a still-pending node. Cancellation may therefore remove a claimed token safely: the already-running signer can finish local computation, but its compare-and-commit becomes unauthorized and persists no certificate issuance or identity. If enrollment commits first, the node is active and cancellation conflicts without mutation. The no-store dashboard validates the exact receipt and, after an interrupted response, claims success only when an authoritative fleet refresh proves the node is absent.

## Immediate host-certificate rotation

The query-free, administrator-authenticated `POST /api/v1/nodes/{nodeID}/certificate/rotate` operation replaces an active enrolled node's host certificate while preserving its pinned Nebula public key. It requires the exact case-sensitive node name, current network configuration revision, and a globally bound 16–128-character persisted request ID. Before signing, the control plane parses exactly one canonical public key from the current certificate and verifies its hash against the pinned identity. The operation is rejected during CA rotation, firewall rollout, or a current certificate-renewal claim.

The `networks.security`-protected, query-free `PUT /api/v1/nodes/{nodeID}/groups` operation is the only ordinary edit path for certificate-bound group membership. It canonicalizes the requested set and always includes `all`, requires the exact node name, current network revision, and a replay-bound request ID, then signs a same-key replacement certificate containing those groups. One atomic commit blocklists the previous fingerprint through expiry, invalidates outstanding agent-recovery records, adds one issuance, advances certificate generation and signed configuration once, and appends an actor-attributed durable receipt. CA, firewall, participating route, and current renewal lifecycles fence the operation. The node-level **Security & access** screen shows current groups, exact effective rules, convergence counters, and recent node audit history before exposing this replacement action.

Signing occurs outside the control-store lock. The final compare-and-commit requires the same certificate, fingerprint, public-key hash, expiry, renewal state, CA/signing material, network revision, and stable lifecycle fences observed before signing. In one transaction Mesh installs a distinct replacement certificate, increments generation once, adds one issuance, blocklists the previous fingerprint through its exact certificate expiry, invalidates every agent-recovery record for that node, advances signed configuration once, and appends the complete non-secret `node.certificate_rotated` audit receipt. A concurrent lifecycle change wins cleanly and the rotation commits nothing.

The request ID makes an ambiguous success exactly replayable. An exact retry returns the persisted receipt without invoking the issuer or writing state; reuse against another node, name, or expected revision conflicts. State-graph validation rejects malformed or duplicate rotation receipts. The dashboard validates the receipt's exact schema and transitions, persists the non-secret request binding in same-tab session storage before sending, automatically retries ambiguous failures, and retains **Verify rotation** across a reload until the receipt is recovered. Rotation may briefly reconnect the managed runtime. It is not a private-key compromise boundary: if the private key is missing or suspect, use identity replacement.

## Node identity replacement

The query-free, administrator-authenticated `POST /api/v1/nodes/{nodeID}/replace` operation is reserved for an active enrolled node whose Nebula private key is lost. It requires the exact current network configuration revision before any mutation. In one control-document transaction it applies the complete ordinary revocation boundary to the source identity, removes every agent-recovery record and relay assignment, safely removes or rolls back any firewall canary selection, blocklists every still-live certificate issuance, advances the network revision exactly once, and creates one pending replacement. The replacement carries the source name, role, groups, site, failure domain, public endpoint, and routed subnets, but receives a fresh node ID and a never-reused IP. It does not carry certificate, public-key, agent-credential, heartbeat, or runtime state.

The replacement enrollment bearer is returned once with `Cache-Control: no-store`, expires after 30 minutes, and is persisted only as a hash. The old routed-subnet reservation is released only inside the transaction after revocation and is immediately reacquired by the pending replacement, so no concurrent route owner can interpose. The route is not published until the new identity enrolls. A stale revision, invalid source state, route conflict, address exhaustion, credential-generation failure, or persistence failure commits neither the revocation nor the replacement.

This operation is deliberately not response-replayable: after commit the source is revoked, so another replace request conflicts instead of creating a second pending identity. If the one-time response is lost, the administrator refreshes authoritative inventory and uses the existing pending-node enrollment-reissue operation on the same-name replacement. Agent recovery is the less disruptive path when only the lifecycle bearer is unavailable and the Nebula private key still exists; it must not be used as a substitute for a lost private key.

## Revoked node archival

The query-free, administrator-authenticated `POST /api/v1/nodes/{nodeID}/archive` operation removes an already-revoked node only after its certificate authority has ended. It requires the exact current network configuration revision and exact case-sensitive node name. For an enrolled identity, the server computes the latest expiry across the node's current certificate, every issuance record, and every revocation record, then refuses archival until five additional minutes have elapsed. Any legacy or permanent revocation entry without an expiry blocks archival permanently. Server time and complete persisted history are authoritative; the dashboard's visible expiry is only a conservative action-presentation aid.

An eligible archival is one control-document transaction. It removes the node, every enrollment and agent-recovery record, every issuance, and every now-expired revocation/blocklist entry, then advances the signed network revision exactly once when blocklist entries were removed. A never-enrolled legacy revoked row has no certificate authority and can archive immediately without changing the revision. A stale revision, mistyped name, non-revoked lifecycle state, incomplete certificate history, unexpired authority, invalid request, or persistence failure commits nothing.

The same transaction appends one strict `node.archived` tombstone with the identity, revocation and expiry times, bounded removal counts, resulting revision, and explicit credential/certificate cleanup facts. State-graph validation rejects duplicate, malformed, prematurely archived, or live-node-conflicting tombstones. Unlike network retirement, the tombstone does not reserve the human name, IP, or routed subnets: reuse is safe only through a fresh random node ID and one-time enrollment credential. Archival does not revoke a node, restore trust, or authorize any old credential; those boundaries were established by the earlier revocation and retained blocklist through expiry.

The no-store HTTP response adds separate runtime-telemetry deletion status because that reconstructible observation store is outside the atomic control document. The dashboard validates the exact receipt and refreshes authoritative inventory before claiming success. If the response is interrupted or malformed, absence proves the control archival committed but does not prove runtime-telemetry cleanup.

## Network retirement

The query-free, administrator-authenticated `POST /api/v1/networks/{networkID}/retire` operation is the hard decommission boundary for one complete Nebula trust domain. It requires both the exact current configuration revision and the exact case-sensitive network name. A stale revision, mistyped name, missing network, invalid request, or persistence failure commits nothing. A successful request atomically removes the network record and its encrypted CA and configuration-signing private keys, every node, enrollment and agent-recovery record, and every certificate issuance and revocation belonging to that network. Old agent credentials, pending enrollment tokens, and recovery tokens therefore become unauthorized immediately.

The same transaction appends one strict `network.retired` audit tombstone containing the retired network identity, CIDR, revision, bounded node counts, removal counts, and explicit credential/key-removal facts. The tombstone permanently reserves the name case-insensitively and rejects any future overlay that overlaps the retired CIDR. This prevents a host holding old Nebula trust material from mistaking a later network for the retired trust and address domain. Tombstones participate in complete state-graph validation; malformed, conflicting, or incomplete retirement evidence causes startup and recovery validation to fail closed.

The HTTP response is `Cache-Control: no-store` and contains the control-plane receipt plus separate runtime-telemetry cleanup status. Runtime observations are reconstructible and live outside the atomic control document, so their deletion occurs only after the retirement commit and is not represented as part of its durability guarantee. The dashboard treats an interrupted or malformed response as ambiguous, refreshes authoritative fleet inventory, and claims completion only if the exact network is absent. Retrying the endpoint after a completed retirement returns not found rather than creating a second tombstone.

Retirement invalidates authority; it cannot synchronously stop an offline or independently supervised Nebula process. Under the packaged fail-closed systemd ownership model, each managed agent becomes unauthorized on its next request and Nebula is quarantined no later than the configured maximum signed-state staleness deadline, five minutes by default. Operators must manually stop fail-open, no-reload, PID/HUP-only, unmanaged, or competing-supervisor Nebula processes before retiring the network. Hosts outside the documented managed-agent and systemd trust assumptions have no enforceable shutdown bound.

After the local preflight passes but before the first enrollment POST, `meshctl` writes a 0600, fsynced provisional journal adjacent to agent state. It contains the one-time token, locally generated bearer, and Nebula keypair so a crash cannot strand a committed node. A restart tries node-scoped authenticated bootstrap before replaying the one-time request and removes the journal only after the signed bundle and final state are durably installed. Resume does not request a second unused-token preflight because the journal proves the point-in-time check already preceded the consuming transaction; current route and DNS state is correlated again through post-enrollment telemetry.

Long-lived node state, its immutable recovery keypair, locks, and every bundle file are private. Output is a dedicated Mesh-owned leaf directory. Complete validated versions are staged outside Nebula's live recursive config scan, fsynced, renamed into `versions/`, and activated through an atomic relative `current` symlink.

## Agent credential recovery

An administrator can issue a separate 256-bit recovery token for an active, non-revoked node. It expires after 30 minutes, is stored by the control plane only as a hash, and is returned once with `Cache-Control: no-store`. Reissuing recovery atomically invalidates unused and in-flight records for that node. A committed signed result is retained so a client that lost its response can still replay the exact old request; the next successful reset prunes every older same-node result. Revocation removes all recovery records.

The recovery token is the sole reset authorization. The request must also carry the node's already-pinned Nebula X25519 public key, but that key is public data: its match binds the reset to the existing node identity and is not proof of private-key possession. Anyone who steals a valid recovery token can replace the agent bearer, deny service, and spoof control-plane health. They still cannot operate the Nebula identity without its private key. Deployments that require a stolen administrative reset token to be insufficient need a separately pinned device-held signing key or mutual-authentication design; the X25519 comparison must not be treated as a second factor.

Before the first recovery request, the node fsyncs both the raw recovery token and a newly generated pending agent bearer into its existing private 0600 state. This intentional node-side secret journal permits `recover-agent --resume` to replay the exact same request after a lost response without asking the operator to retain or re-enter the one-time token; the server never persists either raw value. A fresh administrative recovery supersedes any incomplete normal-rotation bearer, while a resumed recovery must reuse its exact journaled pair.

If a pending token is rejected or expired, the operator may supply an administrator-issued replacement. The client first authenticates with the old pending bearer and verifies its signed bootstrap. A valid old bearer blocks replacement and requires exact resume; only an authoritative HTTP 401 permits an atomic new-token/new-bearer journal. That replacement marker may accept a pinned signed recovery generation strictly greater than local state, rather than exactly local plus one, because an earlier response-lost reset may have committed and then expired while the host was offline. The marker is private, valid only with a pending recovery journal, and is cleared on commit. Every ordinary first attempt and exact resume still requires a generation increment of exactly one.

A signed, domain-separated Ed25519 recovery receipt binds the node, network, new bearer hash, credential generation and expiry, and the exact signed bootstrap digest and signature. The client verifies that receipt only with its previously pinned signer, verifies the complete signed bootstrap and immutable CA/public-key pins, and proves that the pending bearer authenticates before committing it.

Recovery increments the credential generation exactly once and immediately clears both the old current bearer and its grace bearer. A used recovery token can replay only its exact bound result and only while that recovered hash, generation, and expiry remain authoritative; a later normal rotation makes the old recovery result unusable. Rotation, reissue, and revocation races compare against the claimed generation and fail without rolling authoritative state backward.

For the packaged systemd path, `recover-agent --quarantine-service mesh-nebula.service` first verifies the exact managed `ExecStart`, proves the child stopped, and stages without starting it. It rechecks quarantine on exit. The operator starts `mesh-agent.service` only after recovery succeeds, so the ordinary fail-closed startup poll remains the only path that resumes Nebula. If the returned Nebula certificate is already expired, the recovered bearer is committed but the expired bundle is never activated; same-key certificate renewal and validation must succeed while the runtime remains quarantined.

## Signed desired state

Each node artifact binds all of the following under its network Ed25519 key:

- node and network identity;
- network configuration revision and issuance time;
- exact configuration SHA-256 digest;
- exact Nebula CA SHA-256 digest;
- node certificate fingerprint and expiry;
- monotonic node certificate generation;
- exact server-authorized certificate renewal time;
- locally pinned public-key hash.

The agent rejects signature failure, revision rollback, same-revision config equivocation, CA substitution, certificate substitution, public-key substitution, and metadata that does not match locally validated files. It validates a candidate with `nebula-cert verify` and `nebula -test` before activation. Routine certificate renewal changes only that node's signed envelope and ETag; it does not increment the network configuration revision or force peer restarts.

Nebula 1.10.3 is the minimum supported version because older releases are affected by the [P256 certificate blocklist bypass](https://github.com/slackhq/nebula/security/advisories/GHSA-69x3-g4r3-p962).

## Runtime acknowledgment

File activation is not the same as runtime activation. Mesh reports a node online at the desired revision only after exact digest/fingerprint checks and a runtime controller that can make a trustworthy assertion.

The systemd controller requires one bounded effective `ExecStart` record whose actual `path=` and `argv[0]` both equal the expected Nebula binary and whose only arguments are exactly one `-config` and the managed `current/config.yml`. This rejects wrappers, extra flags, duplicate config flags, multiple commands, and systemd's `@` argv spoofing form. Both authenticated units use `Type=exec`, so systemd does not acknowledge the service during its pre-`execve` fork window. Their sole accepted drop-ins are exact installer-owned, comment-only masks for a same-named distribution-wide timeout policy; the effective failure mode must remain `terminate`. The packaged boot contract enables only the Mesh agent. Both managed units require the installer's root-private persistent gate. Nebula additionally requires an ephemeral mode-0400 marker in a systemd-owned mode-0700 `RuntimeDirectory`, which is empty on every boot/agent restart and disappears on agent exit. On startup the agent first removes any interrupted marker and confirms the Nebula child is quarantined, fetches and validates signed state, atomically publishes the marker, and only then starts Nebula. Every stop path removes and syncs the marker before stopping the child. It self-heals a later stopped unit only after another successful signed-config poll. A healthy heartbeat requires one bounded, strictly parsed systemd snapshot reporting `ActiveState=active`, `SubState=running`, and `MainPID>1`; an active-but-exited label is not accepted. Quarantine requires PID zero and an inactive/dead or failed/dead state.

The installer foundation can also stage an exact native bundle into a descriptor-anchored root:wheel tree, publish it below `/opt/mesh/releases` with a no-replace rename, select it through an expected-prior-bound canonical `current` symlink, and atomically publish the exact authenticated root:wheel mode-0644 launchd plist. The native reader requires a physical root:wheel mode-0700 tree with root:wheel single-link mode-0400 files, rejects ACLs, security xattrs, ignored ownership, symlinks, flags, extra entries, truncation, appended data, or identity drift, and streams the signed size/SHA-256 into the same independently rehashed immutable capture. ## Fleet health and telemetry boundaries

Authenticated `GET /api/v1/fleet/health` derives every network and node from one stable control-state read and one server clock sample. Its response is an explicit allowlist: it contains inventory fields, lifecycle phase, typed alert codes with bounded structured evidence, certificate/credential expiry metadata, and rollout counters, but no credential material, certificates, fingerprints, configuration digests, raw agent error text, audit details, or agent-controlled version strings. The read creates no audit event and performs no lifecycle mutation. The per-network `/api/v1/networks/{networkID}/health` projection uses the same derivation rules.

An active node receives a short setup grace before its first heartbeat. At two minutes without an authenticated heartbeat it warns; at five minutes it is critical. A node cannot be operational or rollout-current merely because its counters look plausible: a recent heartbeat must carry an allowlisted agent status, exact current configuration digest and certificate identity, the managed Nebula process must be running, and certificate and agent credentials must be unexpired. Unknown telemetry, future heartbeat/control timestamps, and missing exact proof fail critical. Stale-revocation alerts are emitted only when the latest config-affecting event conclusively matches a live revocation, so unrelated revisions are never guessed to be revocation evidence.

The collection endpoint is the dashboard inventory source, avoiding one request per network. The collection and per-network health routes share a two-request in-process concurrency ceiling; saturation returns a generic `503` with `Retry-After` instead of starting another full-state projection. Browser rendering strictly validates the closed response contract and aggregate arithmetic, uses only typed alert labels, and clears the prior inventory and green state if a request, timestamp, schema, response-timing, or rendering check fails. It refreshes every 15 seconds with one in-flight request at a time, accounts for monotonic header-to-body parsing time against snapshot age, and never recomputes health from browser wall-clock time. `Cache-Control: no-store` applies throughout.

Exact desired-config proof is shared for member nodes and rendered separately for each lighthouse because a lighthouse omits itself from its remote-host list. The projection caps that exact per-network lighthouse work at 64 active lighthouses. A larger or externally corrupted graph returns a typed critical `projection_limit_exceeded` result and does not attempt unbounded quadratic rendering; it never degrades to a guessed healthy result.

`operational` means that the authenticated lifecycle agent recently proved the managed local process and exact signed bundle path. `overlay_reachability_observed` is currently false. The separate packaged observer, versioned store, and aggregate-only fleet endpoint retain observer evidence or explicit `unknown`; consecutive attempted probes may be classified only when their exact heartbeat-authenticated signed-config digest matches. Fleet-health projection does not consume this evidence, and the digest never enters the fleet telemetry projection. Mesh therefore still must not relabel process heartbeats or probe transitions as peer health or general reachability. The accepted boundary is the root-private read-only observer in the [runtime telemetry contract](runtime-telemetry.md); missing observer evidence remains unknown rather than healthy.

## Deployment readiness boundaries

Authenticated `GET /api/v1/networks/{networkID}/readiness` is a separate, no-store, read-only version-6 contract. It uses one stable control-state snapshot to prove Mesh-managed CIDR collisions, configured/active lighthouse counts, canonical site grouping, active-node placement coverage, active-lighthouse failure-domain diversity, endpoint syntax already accepted by the control graph, and the exact lifecycle status present in that snapshot. Topology passes only when every active node has assigned site and failure-domain labels and at least two active lighthouses occupy at least two assigned failure domains. `unassigned` never counts as placement or diversity. The report creates no audit event and does not alter deployment health. Its projection includes at most 64 non-revoked lighthouses; a larger graph is incomplete and blocked rather than truncated into a passing report.

DNS lookup happens after releasing the control snapshot. At most four host-resolver calls run concurrently under one three-second deadline, duplicate names are looked up once, returned values must parse as IP addresses, and only a capped address count is exposed. A successful result means only that the control-plane host's configured resolver answered at that instant. It does not prove that a remote member uses the same resolver, receives the same answer, or can route to that address. Resolver diagnostics and resolved addresses are not returned.

Each current member independently revalidates its active signed bundle, extracts and deduplicates only the lighthouse DNS names in the signed static map, and uses its normal host resolver under a four-worker, three-second bound. Its report contains only version/state, retained sample age, configured-name count, and resolved-name count. Names, returned addresses, resolver configuration, search paths, and errors stay on the node. Member-side DNS passes only when every current active member is operational, exact-heartbeat matched, fresh within 90 seconds, and resolved the exact active-lighthouse name count. One fresh authenticated member failure blocks; partial coverage, stale/future evidence, sequence/config drift, duplicates, unsupported reports, or store failure remains `unknown`.

Client LAN, VPN, VPC, and host routes are not present in the control graph. The supported Linux agent therefore reads the current IPv4 kernel route inventory locally through unprivileged `AF_NETLINK`, identifies the Nebula interface only when it owns the exact address and prefix from the revalidated signed bundle, excludes routes on that interface and the ubiquitous default route, and conservatively treats every other broader, equal, or narrower intersection as a conflict. Its empty capability bounding set prevents route mutation through the netlink socket. The report contains only version/state, server-aged sample age, and one overlap boolean—never interface names or indices, prefixes, gateways, tables, metrics, protocols, or route counts. Non-Linux platforms report `unsupported` until they have an equivalent native proof.

The client-route check passes only when every current active node is operational, exact-config and exact-heartbeat matched, has a unique valid runtime record, and reports no overlap within the 90-second effective freshness window. One fresh authenticated conflict blocks immediately even if other nodes are missing. Partial coverage, stale/future evidence, sequence/config drift, duplicate records, unsupported platforms, or runtime-store failure returns the check to `unknown`; a passing result applies only to current active nodes and does not pre-approve a future machine or site.

Endpoint syntax, DNS success, a UDP `Dial`, and a running local Nebula process still cannot prove public-UDP delivery. That check passes only when every active, operational member has a valid runtime record bound to its exact current heartbeat, the control plane recomputes the complete active-lighthouse target set from each exact currently applied signed configuration and outbound ICMP policy, all active lighthouses are eligible and fit the eight-target probe ceiling, and every member aggregate reports replies from exactly every target. Freshness is derived only from server receive time plus retained sample age and is capped at 30 seconds; missing members, partial replies, stale or future records, sequence/config drift, policy mismatch, duplicate records, store failure, or more than eight active targets return the check to `unknown`. The response exposes only aggregate required/observed member and target counts plus the oldest passing evidence timestamp—never member identity, target addresses, probe plans, packets, nonces, or socket errors.

This is narrowly scoped evidence that one or more current members exchanged authenticated overlay probe traffic with every active lighthouse under the current signed plan. The separate member-DNS check covers only currently active members; neither check clears arbitrary application traffic, future joining sites, or the runtime telemetry fleet API's explicit `end_to_end_reachability_proven: false` boundary. The strict browser adapter rejects unknown fields, inconsistent counts/statuses, fabricated evidence sources, invalid evidence timestamps, duplicate/out-of-order lighthouses, or an overall result that does not match its checks; on request or rendering failure it clears the dialog and infers nothing.

The self-contained Linux public-endpoint smoke exercises the same boundary through two independent routed edge namespaces: each private lighthouse has a default route to its edge, Mesh publishes a distinct test-net UDP endpoint, and an allowlisted nftables DNAT rule forwards only UDP 4242. Positive acceptance requires both DNAT counters plus authenticated dual-lighthouse overlay replies from every member. A site-edge/lighthouse-path loss must evict only that tunnel from the unaffected member, isolate the member assigned to the failed site, and return public-UDP readiness to `unknown`; restoring the same paths must recover the same Nebula processes and all-member evidence. This proves the local routing/NAT/firewall mechanism, not an external provider edge, Internet path, production DNS, or production failure ceremony. While the readiness dialog is open and visible, the browser schedules exactly one follow-up check ten seconds after each completed request; it clears the current presentation before requesting, cancels on close or hidden visibility, resumes with an immediate visible-page check, and closes into sign-in on authentication loss.

## Admin authentication and HTTP exposure

The dashboard uses an HttpOnly, SameSite=Strict cookie containing an independent 256-bit opaque session token, never the administrator bearer. Browser writes require a separate 256-bit CSRF cookie/header value plus exact same-origin checks. Only SHA-256 session/CSRF hashes are persisted. Bearer-based automation does not create a browser session and does not use cookie CSRF. Responses set a restrictive CSP, deny framing, disable content sniffing, and set HSTS when TLS is configured or explicitly terminated by a trusted proxy.

Sessions have bounded idle and absolute expiries, survive process restarts in the private identity store, and can be listed and revoked. Activity may extend the idle deadline but never the absolute deadline. A changed identity-policy fingerprint invalidates older sessions after restart without necessarily marking them revoked; the session inventory can therefore contain expired or policy-stale records that cannot authenticate.

Authentication rejects a wall-clock value earlier than the session's last accepted activity, so clock rollback cannot lengthen an administrator session. A forward jump can expire it early; production time synchronization remains part of the host trust boundary.

When legacy browser login is enabled, the fingerprint is additionally bound to the shared administrator bearer and master key, so a successful restart with a changed binding rejects every browser session, including OIDC sessions. With legacy browser login disabled, administrator-bearer rotation leaves existing OIDC sessions valid. OIDC client-secret changes also leave sessions valid and require restart to load. Replacing the master key is not a supported session-revocation mechanism because existing encrypted CA and signing state also depends on it.

Production listeners must use native TLS or `--behind-tls-proxy`. A proxy-marked backend must bind to a numeric loopback IP; hostnames, wildcard binds, and network-reachable proxy backends are rejected. Other cleartext HTTP is accepted only on an explicit numeric loopback origin, and the deprecated insecure override is rejected. A reverse proxy must be the only path to a proxy-marked listener. `--public-url` is the canonical origin for redirects, Origin checks, and cookie trust decisions.

Production OIDC can run in hybrid or OIDC-only mode. Hybrid retains the shared, full-privilege administrator bearer as the HTTP recovery authority. OIDC-only rejects that bearer over HTTP and instead requires a startup-proven inventory of at least 2-32 usable, one-use break-glass codes. The administrator credential remains required as the offline cryptographic binding for control and backup recovery. Discovery is lazy, so a remote IdP outage does not block hybrid bearer use or OIDC-only break-glass login; invalid local policy, client-secret, identity-state, key material, or usable-code inventory fails startup before either recovery path can be used.

The OIDC path uses Authorization Code with independent one-use state, nonce, and PKCE S256. Strict discovery requires the exact HTTPS issuer, safe endpoints, advertised code/S256 support, an allowed signing algorithm, and an explicit supported client-secret authentication method. Callback verification requires a signed ID token and checks issuer, subject, audience/authorized party, expiry, issued-at, nonce, authentication time, configured ACR/AMR assurance, typed email/group claims, administrator selectors, and `at_hash` when present. Mesh has no UserInfo fallback, rejects `offline_access`, and retains no provider access, ID, or refresh token. MFA is enforced by the configured IdP ACR/AMR claims; one-use break-glass codes are a separate recovery authority, not an MFA factor or native WebAuthn implementation.

OIDC uses a short-lived HttpOnly, Secure, SameSite=Lax transaction cookie so the top-level provider callback can return. Login transactions persist only token hashes plus authenticated-encrypted nonce/PKCE material. A valid state is durably consumed before token exchange; session creation and cookie delivery are later, non-atomic steps. Failure after consumption requires a new login, while a committed session whose response is lost remains inaccessible until expiry or revocation. See [browser identity and OIDC operations](identity.md) for the exact file, IdP, callback, and recovery contract.

## Audit boundaries

Session create/rotate/revoke events commit atomically with their identity-document mutation (`identity-state.json` in the JSON backend). Control lifecycle events commit atomically with their control-document mutation (`state.json` in JSON). Authenticated `GET /api/v1/audit` merges the newest events from both domains into a timestamp-sorted response capped at 100, but the read and the underlying commits are separate. There is no global sequence, integrity link, or external export.

The identity audit is capped at 8,192 durable events and has no pruning or archive path. Reaching the cap fails closed any new session creation or audited rotation/revocation that would require another event. The local logs have no external integrity anchor.

## Backup and restore boundaries

`mesh-backup create` is deliberately offline. It refuses incomplete-restore markers before and after taking the control and identity locks, both nonblocking, before reading either store; a live or starting server and a fenced recovery target all make capture fail. It synchronizes each state file and the data directory, performs stable same-inode reads, strictly decodes both graphs, decrypts every protected value, verifies each Nebula CA certificate's self-signature and matching private key, verifies each configuration-signing pair, validates every current host certificate against its CA/node/issuance metadata, and opens every sealed OIDC attempt. Control schema v2 separately stores scheme-tagged HMAC verifiers for the master key and the administrator credential pair; all complete backup and restore boundaries recompute both. Source identity and bytes are compared after validation, immediately before publication, and once more afterward. Failure is reported as an unsafe snapshot and is never mislabeled as a publication-durability problem. It encrypts the exact state bytes plus canonical master/admin credential files under a separate 256-bit backup key. Archive paths are create-only and publication uses a synchronized/reopened private temporary inode plus a no-replace hard link. `verify` can repair a crash that leaves final and one exact random temporary name linked to the same authenticated inode; missing, ambiguous, or nonmatching candidates fail without unlinking.

The master verifier lets a reader of `state.json` test candidate master keys offline. Mesh therefore requires a true 256-bit CSPRNG master key and does not support padded or directly password-derived 32-byte inputs. A read-only startup preflight rejects bound credential mismatches before compatibility migrations. Administrator rotation requires the one-shot `--rotate-admin-token` authority, never permits a master-key change, and records `admin.credential_rotated`; retaining that flag after the intended startup weakens typo protection. The version-2 state change is one-way for older strict decoders. Live version-1 state can be bound by one successful upgraded startup, but current backup/restore deliberately rejects unbound version-1 archives.

Archive authentication uses an exact versioned header, HKDF-SHA-256 key separation with a random 256-bit salt, and AES-256-GCM with a random nonce. The complete header, including the ciphertext length, is authenticated as additional data. Framing, manifest JSON, fixed entry order, modes, lengths, digests, and credential encodings are strict and bounded. There is no password-derived mode, compression parser, unencrypted manifest, or partial-stream recovery surface.

Restore requires a caller-supplied authenticated backup ID and a nonexistent target under a private real parent. A durable sibling marker fences the target before its directory is created. Each file and receipt is created with no replacement, reopened, cryptographically revalidated, synchronized, and accepted only as the exact expected directory set. On Linux, restore/finalization and `mesh-server` startup take the same nonblocking advisory lock on the opened parent directory. Restore holds it from before target/marker inspection through marker removal or failure; the server holds it across its alias-aware precheck, control-store creation/lock, and postcheck. Exact markers fence absent targets. Existing targets additionally trigger a bounded sibling scan whose marker-derived candidates are compared by inode, so case-folded and same-parent alternate spellings cannot bypass the fence; scan overflow or errors fail closed. The advisory lock does not constrain unrelated software. An interrupted operation can remove the marker only through `finalize-restore`, which repeats every file and directory durability barrier and validates the receipt and state first.

The restore receipt HMAC is keyed from the recovered master key and includes both canonical recovered credential bodies. It detects accidental corruption and incomplete operator handling. It is not an external integrity anchor: a local attacker who can read the restored master key can forge the receipt, and an older self-consistent archive includes its matching key and passes every local check. `--expect-backup-id` prevents selection mix-ups, not rollback. Authorization of the recovery point requires an independently protected monotonic catalog containing the expected ID, sequence, archive digest, and key-custody reference.

The current command mutates files only on Linux, where it proves effective-user ownership, owner-only modes, single-link regular files, symlink-free path components, advisory locking, and directory synchronization. macOS and Windows backup/restore fail closed because ACL, case-folding/DACL, advisory-lock, and directory-durability guarantees have not all been established. Non-Linux servers retain alias-aware marker checks, but no local restore command exists there to race startup. See [backup and recovery](backup-recovery.md) for the complete operator ceremony and live proof.

`mesh-storage import-backup` reuses the Linux archive reader and all recovery-grade credential/CA/certificate/signing/OIDC validation, then rereads and revalidates the exact PostgreSQL pair before declaring success. `mesh-storage verify` must run offline with every writer stopped. Neither command exports a later PostgreSQL state or replaces database base-backup/WAL/PITR operations.

## Network policy defaults

New nodes receive the `all` group. A new network starts in managed-policy mode with the historical compatibility policy: authenticated `all` members are permitted inbound and any destination is permitted outbound. Stores written before policy management are decoded into an explicit `legacy-default` compatibility mode with those exact effective rules. This gets a first network connected quickly, but it is not least privilege; operators should replace it after enrollment.

The dashboard and authenticated admin API expose a deliberately narrow subset of Slack Nebula v1.10.3 firewall rules. Each rule has one exact protocol (`any`, `tcp`, `udp`, or `icmp`), one canonical port or ascending range, exactly one remote selector (wildcard, certificate group, named same-network node, or canonical in-network IPv4/CIDR), and at most one local installation target (a named node or certificate group). An empty local target installs the rule on every node. Named peer nodes are resolved to their current overlay `/32` during per-node compilation; local targets are removed before Nebula YAML is rendered. Arbitrary certificate-hostname selectors, IPv6, unsafe-route selectors, fragments, CA selectors, multi-group conjunctions, and Nebula's surprising `group: any` wildcard are not accepted. ICMP requires `port: any`. Empty inbound or outbound arrays are rendered explicitly and deny that direction by default.

Each direction is capped at 128 rules and the aggregate concrete port expansion is capped at 16,384 slots before a config can be stored. Rules are canonicalized, sorted, and rejected if two inputs are semantically equivalent, including an IPv4 address and its `/32`. Preview returns the source-policy digest plus the exact compiled rules, rendered Nebula YAML, and digest for every active node. A deployment supplies the previewed configuration revision; a different effective policy cannot overwrite a newer revision, while an exact lost-response retry remains idempotent. One effective change advances the network revision once, records old/new source-policy SHA-256 values and bounded counts in the same atomic state transaction, and changes every affected node's signed desired config.

Control schema v13 is the one-way compatibility boundary for these scoped fields. Migrating a v12 document appends one audit event but changes no network policy, signed bytes, configuration revision, or timestamp. A scoped policy cannot be stored before v13, and older binaries reject the new control version instead of silently discarding rule fields.

Routed-subnet ownership is deliberately narrower than general Nebula unsafe-route configuration. A node may declare at most eight canonical, non-default IPv4 unicast prefixes when it is created. Pending and active owners reserve those ranges across every managed overlay and live routed subnet; partial, covering, cross-network, managed-CIDR, and special-range overlap is rejected, while revocation releases the reservation. Creating an ordinary pending duplicate is also rejected. The route-profile workflow may add an exact prefix already served by active enrolled owners in the same network, with at most eight owners. Atomic identity replacement is the narrow pending exception: after revoking an already-authorized owner route-first, its pending successor may retain the exact prefix only while another active enrolled same-network owner serves it; the successor is not published until enrollment. Enrollment and renewal embed the exact list in each owner certificate. Only active owners are published to peers, every owner's own configuration omits the locally served route, and each gateway's managed inbound rules are rendered again with every exact routed subnet as `local_cidr`.

The query-free administrator route-transfer API can move exact prefixes between distinct active enrolled nodes in the same network. Start is bound to the visible configuration revision, exact prefix set, source, target, and a 16–128-character request ID. It advances one signed revision, renders the staged prefixes only into the target's local firewall expansion, and authenticates `certificate_profile_renewal_required` in a new signed-artifact version. Legacy agents cannot validate that artifact and fail closed. Peers continue routing through the source. The target renews its immutable public key with the union certificate profile; promotion remains unavailable until a fresh healthy heartbeat proves the current authoritative certificate generation and fingerprint, exact current config revision and digest, and running Nebula.

Promotion atomically removes the exact prefixes from the source record, adds them to the target, advances one signed revision, and changes peer unsafe routes to the target. It then forces source certificate renewal without the moved prefixes. Completion changes only the durable receipt and audit after exact source convergence; it does not invent another config revision. Before target issuance, cancellation removes staging in one signed revision. After issuance, cancellation enters `cleaning_target`, forces a certificate back to the original profile, and remains incomplete until exact target convergence. No timeout promotes or cancels. Reads always expose the authoritative receipt for response-loss recovery. One transfer may be active per network.

The node-scoped route-profile API replaces one active enrolled gateway's complete exact routed-prefix set. Start is bound to the visible configuration revision, final set, node, and a 16–128-character request ID. Additions are certificate-first: the owner receives the union of original and final prefixes through the signed mandatory-renewal artifact while peers keep only the original routes. Promotion remains unavailable until an exact healthy owner heartbeat proves the staged generation, fingerprint, signed revision and digest, and running Nebula; only then does one atomic signed revision publish the final peer route set. Add-only edits complete at promotion.

Removals are route-first. A removal-only start immediately withdraws the removed peer routes in one signed revision and forces the owner to renew a certificate containing only the final set. Mixed edits enter that cleanup phase after certificate-first promotion. Completion remains unavailable until the exact cleaned certificate and current signed config converge. Cancellation is available only before promotion: before staged issuance it clears prepare in one revision; after issuance it forces the certificate back to the original set and requires exact cleanup convergence. Terminal advance/cancel replays accept only the current or immediately preceding bound revision and write no state. Final additions are reserved graph-wide during prepare. An old/new partial overlap or a staged union above eight prefixes cannot be represented by this safety sequence and fails closed; complete the removal first, then start the addition.

For an exact prefix with multiple active owners, Mesh renders one Nebula `unsafe_routes` entry with a deterministic list of `{gateway, weight}` objects; duplicate entries would overwrite in Nebula's prefix table and are never emitted. The administrator-authenticated route-policy API requires the complete server-derived owner set, the visible signed revision, a 16–128-character request ID, weights from 1 through 1,000, MTU 0 or 500–65,535, and metric 0 through 2,147,483,647. Mesh always installs the managed route. Exact response-loss replay is write-free; stale or incomplete owner sets fail closed. A departed owner is removed route-first, surviving weights/MTU/metric remain, and a new certificate-first owner starts with weight 1. Nebula hashes transport-port pairs by relative weight. If the selected gateway has no ready tunnel, Nebula tries a ready alternate without recalculating weights; this is connectivity fallback, not health-aware rebalancing or a router SLA. Route transfer remains single-owner only.

Managed firewall renderer v2 double-quotes every certificate-group scalar before Nebula parses YAML. This prevents valid group names such as `False`, `null`, `001`, or an ISO-style date from being resolved as booleans, nulls, numbers, or timestamps before Nebula stringifies the rule. Historical `legacy-default` policies retain their exact v1 bytes. On startup, an explicitly managed v1 policy is upgraded atomically: a byte-changing upgrade advances its signed network revision, timestamp, and hash audit exactly once; a group-free byte-identical upgrade records only the renderer version; revision exhaustion aborts the whole migration.

The live Linux namespace/TUN proof validates each generated bundle with exact-version Nebula 1.10.3 binaries. Its firewall-rollout mode first proves that exact authenticated canary activation-failure evidence automatically restores the retained policy while authenticated traffic and both original Nebula processes survive; a second target run proves the exact stopped-runtime heartbeat guard does the same without exposing its raw diagnostic. It then proves TCP/443 crosses under the restrictive canary policy, ICMP is denied from clean conntrack state, pause restores the retained policy and ICMP, resume re-applies the target in a fresh signed revision with new convergence evidence, manual rollback restores delivery, and exact convergence gates fleet-wide promotion. Its routed-subnet mode additionally verifies the gateway certificate and both signed configurations, then sends real packets from a member through the gateway to a separate non-Nebula namespace. Revocation removes the published route. The route-profile variant withdraws that route before certificate cleanup, proves packets stop, restores certificate authorization while the peer route remains unpublished, and proves promotion restores packets without replacing either Nebula process. The ECMP variant adds a second owner certificate-first, proves bounded 3:1 UDP delivery through both gateways, underlay-loss fallback after Nebula tunnel-death detection, weighted recovery, and route-first membership removal while all three original Nebula processes survive. This harness checks version and behavior; the separate dependency-intake path is what authenticates locked upstream bytes. Broader configurable or multi-sample health rollback, production router failure/SLA work, and per-service templates remain separate work.

Every rendered config explicitly sets `pki.disconnect_invalid: true`. Revocation records every still-valid issuance for the node and publishes the complete fingerprint blocklist in the next signed artifact.

## Revocation guarantees and limits

Revocation immediately disables the node's current and grace-period agent credentials, blocks renewal/bootstrap, records all unexpired issued fingerprints, and advances the network configuration revision. Online systemd-managed peers poll, validate, restart onto the new blocklist, and acknowledge the exact config revision and certificate generation.

The dashboard uses the query-free, administrator-authenticated `POST /api/v1/nodes/{nodeID}/revocation` trust-cutoff endpoint. Its body requires the exact case-sensitive node name, current network configuration revision, and a 16–128-character request ID. One control-document transaction marks the node revoked; clears current and grace agent credentials and renewal claims; removes all enrollment and agent-recovery records; records every applicable certificate fingerprint; releases routed-subnet ownership; removes relay assignment; removes the node from any firewall canary and rolls the staged policy back if it was the final canary; advances signed configuration once; and appends an actor-attributed `node.revocation_committed` receipt. No raw credential, certificate, or fingerprint appears in that receipt.

An exact request retry returns the persisted receipt without writing state. Reusing the request ID for another node, name, or expected revision conflicts, and state validation rejects malformed or duplicate receipt events. Before sending, the browser stores the non-secret binding in same-tab session storage; ambiguous failures retain **Verify revocation**, including across reload, while a strict receipt plus authoritative revoked inventory and revision readback is required for a normal success claim. The older `/revoke` route remains a compatibility surface for existing automation, but the managed dashboard uses only the exact replayable contract.

Production recovery credentials may be supplied through clean absolute `--admin-token-file` and `--master-key-file` paths instead of process-environment values. Each path is traversed without accepting symlinked directory components; the leaf must remain one bounded, single-link, owner-controlled regular file with no group/other permissions before and after opening. The value must be one printable-ASCII credential with only an optional final newline. Supplying the corresponding environment variable at the same time is rejected. The container contract uses only these file paths, so neither raw recovery credential appears in the image configuration, Compose environment, or process arguments. Native Windows file credentials remain unavailable until a package can prove their DACLs.

The reference Compose service terminates TLS inside Mesh, runs the scratch image under an explicit non-root UID/GID, mounts only the JSON state directory read-write, mounts credentials and TLS material read-only, drops every capability, enables `no-new-privileges`, and makes the remaining root filesystem read-only. Its health command trusts an explicit CA bundle, connects only to an explicit loopback-IP `/readyz` URL, verifies the configured certificate DNS name/IP, rejects redirects and inexact/oversized JSON, and succeeds only when the server's authoritative control, identity, credential-binding, and runtime-telemetry checks pass. This is a local single-replica mechanism, not a signed image, HA deployment, host/Docker hardening proof, ingress policy, backup policy, or node-installer trust channel.

The Helm contract preserves that private-file boundary rather than passing recovery credentials through environment variables or teaching the server to accept Kubernetes Secret symlinks. One same-image init command runs before the server with only `CHOWN`, `DAC_OVERRIDE`, and `FOWNER`; it resolves each fixed projected key only inside its mounted source root, takes a bounded stable snapshot, validates both recovery credentials and the certificate/key/CA/hostname chain, and publishes an exact memory-backed mode-0700 directory of single-link files owned by UID/GID 65532. Optional OIDC and PostgreSQL files use the same fixed inventory. The main container receives the private volume read-only, runs with no capabilities or service-account token, and verifies `/readyz` over loopback TLS. JSON mode is one `Recreate` replica on one PVC; multiple rolling application replicas require the external PostgreSQL authority. The chart's image digest, Secret names, public HTTPS origin, storage mode, replica count, and optional edge features are schema-gated. This does not authenticate the registry, ingress controller, Secret/PVC implementation, Kubernetes nodes, admission path, PostgreSQL deployment, or backup/restore operations; those remain target-cluster proof obligations.

`mesh-origin` is a separate public process with no Mesh administrator, node, CA, root-role, or release-role private credential. Its canonical index is an explicit object allowlist, not an authority: startup rejects unknown index fields, noncanonical paths/order/JSON, duplicate objects, invalid cache/content classifications, symlinked directory components, hard-linked leaf files, size drift, SHA-256 drift, and a noncanonical stable channel bundle. Linux publication copies only those objects into a content-addressed generation, checks every source and destination byte, writes a canonical operational receipt, fsyncs and seals the exact tree, and uses a no-replace rename. Inspection rejects extra paths, writable or linked files, receipt/index mismatch, and object drift. Compose derives repository and index mounts from one generation path, so rollout and rollback cannot combine unrelated halves. The read-only external auditor starts from that inspected local generation, disables environment proxies, cookies, redirects, compression, and HTTP/2, requires one consistent verified TLS leaf, checks exact readiness/security and negative-route behavior, and performs HEAD plus full bounded digest-checked GET verification for every object before emitting a create-only receipt. Only exact GET/HEAD requests without encoded paths or queries are served; there is no listing, redirect, range mutation, upload, compression, or content negotiation. Open descriptors and pre/post-stream metadata checks preserve the selected generation, while an in-place change fails inspection, external audit, readiness, and transfer. The origin may courier the canonical root, root-authorized bootstrap manifest and signatures, installer, both deterministic standalone-verifier packages, and unsigned canonical handoff, but it authenticates none of them. The preferred authority is a canonical create-only bootstrap anchor transferred through a separately controlled channel and kept off the origin. The narrow verifier reads that anchor before the courier handoff, binds its exact bytes and repeated review fields, and only then derives the selected package and root digests. Direct handoff or root digests remain lower-level alternatives. The narrow verifier, not the origin, establishes threshold installer authority without execution. The signed installer remains responsible for root-chain, threshold, epoch, replay, platform, exact-size, and artifact-digest authorization during subsequent release installation.

The origin container is a read-only, capability-free scratch runtime under an explicit non-root UID/GID with only its public repository, canonical index, and TLS material mounted read-only. Native TLS keys must be bounded, single-link, owner-only files and are read through stable descriptors before serving. Production Compose contains no build or mutable image default: it requires one registry repository plus exact manifest SHA-256. The separate origin image-security gate authenticates every saved-image object, its exact 18-path Linux amd64 scratch filesystem and metadata, exact Syft/SPDX package inventory, current Grype policy, and empty rootfs/binary-string Gitleaks reports before writing a create-only canonical receipt. `mesh-origin-image-verify` separately snapshots an independently provisioned Cosign public key, explicitly checks Cosign claims for the published digest, rehashes the bounded Cosign executable, and writes create-only signature evidence only after every returned payload binds the requested digest. After start, `mesh-origin-runtime-verify` strictly reparses both receipts and binds their hashes to one production Compose render with no build, one fully inspected generation, one exact healthy 64-character Docker container ID, the security receipt's exact local Docker ID and Linux amd64 platform, the authenticated repository digest, non-root command/user, runtime hardening, and read-only mounts. It forces one explicit real Unix socket and exact Docker executable, rechecks both plus the generation, and mutates no runtime state. The local build exists in a separately named override that must not be used in production. The repository does not publish or sign the real image, distribute the public key, install Cosign, retain receipts, or provide registry admission control.

Host or Docker authority can still lie to the runtime verifier or replace the entire container, certificate, repository, or index after it runs; clients remain protected from software substitution by independent installer trust. Security, signature, and runtime receipts prove point-in-time local observations, not remote attestation or continuous admission, and newly disclosed vulnerabilities can stale retained scan evidence. The external auditor independently detects a public-route mismatch only when it runs and its receipt/exit status reaches a trusted monitor. Origin availability, alert delivery and receipt retention, image/public-key custody, runtime admission/attestation, rollback policy, DNS/TLS operations, and real multi-custodian key ceremonies therefore remain operator responsibilities.

The authenticated install guide may expose a canonical HTTPS bootstrap-handoff courier URL independently of online release availability. It labels that URL as location only, never accepts or returns a bootstrap anchor or expected handoff digest, and does not generate a pre-trust execution script. A compromised control plane, origin, or TLS route can therefore redirect or deny the courier but cannot supply the independently transferred authority required before trusting the verifier package or root.

With the packaged systemd ownership model and default flags, a running peer cannot continue using a config whose last successful signed-state confirmation is more than five minutes old. On reboot, the ephemeral child gate is absent and Nebula remains stopped until the first successful sync. This bound assumes the agent and systemd control path remain operational, the host clock does not jump forward/backward outside the fail-closed checks, OS root and root-owned systemd configuration remain trusted, no competing supervisor bypasses the managed unit, and operators have not selected `--fail-open`.

An offline host cannot receive a new blocklist, but it also cannot exchange traffic while powered off; its packaged boot path gates Nebula on fresh state when it returns. PID/HUP peers never claim positive runtime application, and no-reload/fail-open peers have no enforcement bound. Telemetry-to-alert classification, handshake freshness policy, quorum gates, and enforcement-SLO monitoring remain outstanding; the [runtime telemetry contract](runtime-telemetry.md) records the accepted local-observer boundary and rejects Prometheus/SSH shortcuts. Operators must treat any non-acknowledging or fail-open peer as potentially stale.

## Current operational limits

- Single-process JSON persistence is atomic but not highly available or suitable for multiple writers. The explicit PostgreSQL backend supports multiple application processes and has bounded local promotion, intended-workload micro-soak, archived-WAL point-in-time recovery proofs, and a first audited maximum-valid-document pass. Production database HA, independently custodied recovery authorization, archive retention, timeline operations, sustained fault injection, and measured production RPO/RTO remain outstanding.
- Credential lookup remains linear and each real mutation rewrites the full control or identity document (a JSON file or PostgreSQL `BYTEA`). The current model is for small deployments, not fleet scale.
- Coordinated encrypted backup, create-only restore, crash fencing, and a real-Nebula restore drill are implemented. Local validation still cannot distinguish a self-consistent old archive from the authorized recovery point; rollback protection requires the external monotonic catalog described above.
- A threshold-authenticated Linux installer now owns immutable releases, exact systemd units, activation, upgrade, recovery, and rollback from either a root-private offline snapshot or a canonical HTTPS release bundle. Its compiled version-1 root and append-only dual-threshold root chain support crash-durable signing-key rotation/revocation and release epochs before artifact retrieval, with privileged re-verification after download. A narrow non-signing `mesh-bootstrap-verify` binary statically validates the independently rooted first installer; deterministic create-only USTARs bind each Linux verifier's production identity and exact bytes without including authoring tools. An unsigned canonical handoff revalidates and binds both packages and root version 1, while a second unsigned create-only anchor makes that authority independently transferable and human-reviewable. The verifier reads the anchor first and emits a v3 receipt binding the full chain; direct handoff/root digest compatibility emits v2/v1. The real independent transfer channel, image publication, and custodian ceremony remain external. Windows staging USTARs bind source-authenticated security-patched Nebula PEs plus exact upstream Wintun bytes; Darwin staging USTARs bind source-authenticated thin Nebula Mach-O executables plus reviewed launchd assets. Both pass exact final-candidate SBOM/vulnerability/secret gates and require matching release-authoring receipts. The Darwin agent also contains a descriptor-anchored [native path-security adapter](darwin-path-security.md) whose parser, fault-injected walk, and both target builds are proved on Linux. These remain deliberately non-installing foundations: a real Mac must still prove the ACL syscall and race boundary, while native package/code signing, Windows DACLs, launchd/Windows Service activation, notarization, and installed-host validation remain outstanding.
- Planned online CA rotation is implemented, but emergency compromise recovery, subordinate/offline-root mode, HSM/KMS custody, multi-party approval, and disaster-recovery ceremonies are not. An attacker with the active online CA or master key is outside the planned-rotation safety model.
- Agent recovery follows a password-reset trust model: possession of the short-lived administrator recovery token is sufficient to replace the agent bearer when the pinned public identity is supplied. A device-held proof-of-possession recovery factor is not implemented.
- Node bundle retention is capped at five validated immutable versions. Explicit revoked-node archival removes one eligible node's credential and certificate histories after their authority expires, but there is no automatic retention policy, bulk archival, bounded control audit, or bounded history for records that remain live; the separate identity audit instead has the hard fail-closed cap described above.
- Pre-release node state schema v2 intentionally rejects prototype v1 state and journals; a supported migration path is required before a stable release.
- Pre-release Linux installer state schema v3 binds epoch, accepted root, bootstrap root, package identity, and verification time. It migrates an exact quiescent v2 state only under the untouched compiled root with empty root history; prototype v1 pending journals remain unsupported. Linux bundle v3 binds a candidate installer-only read range and write version to the exact `mesh-install` ELF frame, and staging rejects an inherited schema outside that authenticated range before publication. Legacy bundle v2 is bridged only to state v3 and is rejected for any future state version.
- Windows file privacy ultimately depends on installer-created DACLs for the service account because Go file modes do not express them. The current native Windows `mesh-server` fails closed even in legacy-only mode because it cannot prove the identity-state DACL; any hybrid policy and client secret also fail closed.
- OIDC and agent pre-authentication budgets are process-local and memory-bounded; an internet-facing deployment still needs trusted-edge distributed controls. Behind the loopback TLS proxy, every OIDC request shares the proxy's address bucket because Mesh does not trust forwarded client-IP headers. The legacy browser-token form has bounded delayed-failure concurrency but no token bucket.
- Development HTTP cookies are host-scoped, not port-scoped. Loopback browser identity therefore trusts every local web service on the same numeric loopback host not to capture and replay Mesh cookies; use an isolated development host/browser profile when that assumption is not valid.
- Native WebAuthn, scoped administrator roles, tenant isolation, and production multi-operator custody ceremonies for implemented one-use break-glass codes are not implemented or proved.
- Fleet health, typed lifecycle alerts, lighthouse redundancy, stale-revocation detection, and rollout convergence are implemented from authenticated agent heartbeats. They prove the managed local process/config path, not recent peer handshakes or overlay packets. Runtime telemetry transport/persistence and its authenticated aggregate-only API/dashboard projection are separate: the UI requires an exact heartbeat binding, omits process identity and the private config digest, and never calls an observation healthy or generally reachable. Packaged passive continuity and config-bound probe-transition classification are implemented; promotion into lifecycle health or broader automatic alert/enforcement policy remains outstanding under the fail-closed [runtime telemetry contract](runtime-telemetry.md).
